## 0.5 배포 방식 결정 — Path A(셀프서비스, 기본) vs Path B(중앙 강제배포, 옵션)

> **본 환경 확정: 개발자 모드 활성화 가능 → Path A(셀프서비스)가 기본.** 아래 §1의 `key` 고정·`.crx`·강제설치는 **Path B(중앙 강제배포) 옵션**이며, 중앙 자동 업데이트·비기술 사용자 확대·확장 무결성 중앙 보증이 필요할 때만 채택합니다.

### Path A — 셀프서비스 (본 환경 기본 경로)

원본 `install.sh` + README 절차가 **애초에 셀프서비스용으로 설계**되어 있습니다. `install.sh`가 확장 ID를 **인자로 받는** 이유가 바로 이것 — PC마다 ID가 달라도 각자 자기 ID로 등록하면 됩니다.

**중앙에서 1회 준비(공유 폴더/패키지로 배포):**
- vendored `node_modules`(`.tgz`, 에어갭이라 `npm install` 불가 — `02` 문서 §2.2)
- Node v20 LTS 런타임, 사내 표준 Chromium 브라우저, Claude Code 오프라인 패키지
- LLM 게이트웨이 env(`ANTHROPIC_BASE_URL` 등) 표준값
- install 스크립트: mac/Linux `install.sh` / **Windows `install.ps1`(신규 작성 필요, §2.2)**

**각 사용자 셀프 절차:**
1. 확장/호스트 폴더를 표준 경로에 복사하고 `host/node_modules` 복원(`.tgz` 압축 해제)
2. `chrome://extensions` → 개발자 모드 ON → **Load unpacked** → 확장 ID 복사
3. 자기 ID로 install 실행: `./install.sh <내-확장-ID>` (Windows: `.\install.ps1 -ExtensionIds <내-ID>`) → 네이티브 호스트 등록
4. 브라우저 완전 재시작(전 창 종료)
5. `claude mcp add open-claude-in-chrome -- node <경로>/host/mcp-server.js`

**유의:** 일부 브라우저는 시작 시 "개발자 모드 확장을 사용 중지하시겠습니까?" 경고를 노출 — 셀프서비스에선 사용자가 "유지"를 선택하면 됩니다(이 경고/핀 고정을 정책으로 억제하려면 Path B).

### Path A vs Path B 비교

| 기준 | Path A — 셀프서비스(기본) | Path B — 중앙 강제배포(옵션) |
|---|---|---|
| 전제 | **개발자 모드 허용**(본 환경 ✅) | 개발자 모드 차단 정책이거나 중앙 통제 요구 |
| 확장 ID | PC별 상이 OK (install이 인자로 수용) | `key`로 **고정** 필요 |
| 설치 | 사용자가 Load unpacked + install 실행 | GPO/MSI가 `.crx` 강제설치 |
| 업데이트 | 사용자가 폴더 갱신 후 reload | `update.xml` 버전 범프 자동 전파 |
| 적합 | **소수~중간 규모 기술 인력**(Claude Code 사용자) | 비기술 사용자 대량·중앙 감사·자동 업데이트 |
| 작업량 | 낮음(스크립트·가이드) | 높음(키·서명·update 서버·정책) |

**언제 Path B로 전환?** ① 개발자 모드가 정책으로 차단될 때(사용자 수 무관·필수) ② 비기술 사용자로 확대 ③ 중앙 자동 업데이트 필요 ④ 시작 경고 제거·핀 고정 강제 ⑤ 위장 확장 차단(무결성 중앙 보증).

> **배포 방식과 무관하게 공통 필요:** Windows 지원(`install.ps1`·wrapper·레지스트리, §2.2), vendored `node_modules`·Node 반입, 그리고 **보안 하드닝(`03` 문서 High 항목)**. 셀프서비스는 중앙 통제가 약해 오히려 감사 로깅 중요도가 올라갑니다.

## 1. (Path B) 중앙 강제배포 상세 — 옵션

> 아래는 **Path B(중앙 강제배포) 옵션**입니다. 본 환경은 개발자 모드가 허용되어 **Path A(셀프서비스, §0.5)가 기본**이며, 위 "언제 Path B로 전환?" 조건에 해당할 때만 적용하세요.

### 1.1 (Path B 한정) 개발자 모드 배포의 한계 — 강제배포가 필요한 경우

현재 `install.sh`가 **확장 ID를 인자로 받는 이유**가 바로 이 문제입니다.

> "Each Chromium browser assigns a different ID when loading unpacked extensions" (install.sh 주석)

"Load unpacked"로 로드하면 확장 ID는 **로드된 디렉터리의 절대 경로 기반 해시**로 생성됩니다:
- PC마다 설치 경로가 다르면 → ID가 달라짐
- 같은 경로라도 브라우저(Chrome/Brave/Edge)마다 → ID가 달라짐
- ID가 달라지면 → 네이티브 호스트 manifest의 `allowed_origins`(`chrome-extension://<ID>/`)가 **PC마다 달라져야** 함 → 표준화 불가능 → 대규모 배포 붕괴

추가 문제:
- Chrome 시작 시 "개발자 모드 확장을 사용 중지하시겠습니까?" 경고/자동 비활성화
- 정책으로 개발자 모드 자체를 막는 보안 표준과 충돌
- 자동 업데이트 불가

**결론: 개발자 모드는 PoC 단계에서만. 운영 배포는 반드시 `.crx` + 강제 설치 정책.**

### 1.2 선결 작업 — 확장 ID 고정 (manifest `key`)

확장 ID는 `key`(공개키)의 SHA-256 해시 앞 128비트를 a–p 문자로 인코딩한 값. `key`가 고정되면 ID도 고정됩니다.

```bash
# 1) 확장용 키페어 1회 생성 (오프라인 빌드 머신에서 안전 보관)
openssl genrsa -out open-claude.pem 2048

# 2) 공개키(DER, base64) 추출 → manifest의 key 값
openssl rsa -in open-claude.pem -pubout -outform DER 2>/dev/null | openssl base64 -A
```

`manifest.json`에 추가:
```json
{
  "manifest_version": 3,
  "name": "Open Claude in Chrome",
  "key": "MIIBIjANBgkqhki...(위 2단계 출력)...AQAB"
}
```

> 실무에서는 키페어로 한 번 .crx를 만들어 `chrome://extensions`에 로드해 ID를 확인하는 것이 가장 확실합니다. 이 ID를 **사내 표준 상수**로 고정합니다.

**효과:** 이후 `install.sh`/Windows 등록 스크립트의 `allowed_origins`를 **고정 단일 값**으로 박을 수 있어, PC별 분기/인자 입력이 사라집니다. 같은 코드라면 Chrome/Edge/Brave 모두 `key` 기반 **동일 ID**(단, 정책 키 경로는 브라우저마다 다름 — §1.4).

### 1.3 자체 서명 .crx 패키징 + 내부 업데이트 서버

**(A) .crx 패키징** (오프라인 빌드 머신):
```bash
# 방법 1: Chrome CLI 패키징 (key.pem 재사용으로 ID 고정 유지)
google-chrome --pack-extension=/build/extension \
              --pack-extension-key=/secure/open-claude.pem
# → /build/extension.crx 생성

# 방법 2: CI에서 crx3 도구 사용
npx crx3 -p /secure/open-claude.pem -o open-claude-1.2.0.crx /build/extension
```

**(B) 업데이트 매니페스트** (`update.xml`) — 사내 서버에 게시:
```xml
<?xml version='1.0' encoding='UTF-8'?>
<gupdate xmlns='http://www.google.com/update2/response' protocol='2.0'>
  <app appid='<고정확장ID>'>
    <updatecheck codebase='https://ext.corp.local/openclaude/open-claude-1.2.0.crx'
                 version='1.2.0' />
  </app>
</gupdate>
```

**(C) `manifest.json`에 update_url 추가:**
```json
{ "update_url": "https://ext.corp.local/openclaude/update.xml" }
```

**(D) 내부 업데이트 서버 요건:**

| 요건 | 내용 |
|---|---|
| 프로토콜 | HTTPS (사내 CA 서명 인증서, 모든 PC에 루트 CA 신뢰 배포 필수) |
| MIME | `.crx` → `application/x-chrome-extension`, `update.xml` → `application/xml` |
| 가용성 | 정적 파일 서버(nginx/IIS) 2대 + L4 LB. 업데이트 체크는 5시간 주기라 부하 낮음 |
| 버전 관리 | 디렉터리에 버전별 .crx 보존(롤백용), `update.xml`만 교체하면 전파 |

### 1.4 그룹 정책 / 레지스트리 강제 설치

#### ExtensionInstallForcelist (강제 설치 + 자동 업데이트 + 비활성화 불가)

**Windows 레지스트리 (Chrome):**
```
경로: HKLM\SOFTWARE\Policies\Google\Chrome\ExtensionInstallForcelist
값(REG_SZ): 1 = "<고정확장ID>;https://ext.corp.local/openclaude/update.xml"
```

**브라우저별 정책 하이브:**

| 브라우저 | 레지스트리 경로 |
|---|---|
| Google Chrome | `HKLM\SOFTWARE\Policies\Google\Chrome\ExtensionInstallForcelist` |
| Microsoft Edge | `HKLM\SOFTWARE\Policies\Microsoft\Edge\ExtensionInstallForcelist` |
| Brave | `HKLM\SOFTWARE\Policies\BraveSoftware\Brave\ExtensionInstallForcelist` |

> **단일 브라우저 표준 권장.** 에어갭 운영 복잡도를 낮추려면 사내 표준 브라우저 1종(예: Edge 또는 Chrome)만 강제설치 대상으로. 멀티 브라우저는 네이티브 호스트 manifest를 브라우저별 디렉터리에 중복 배포해야 함.

#### ExtensionSettings (정밀 제어 — 권장)

```
경로: HKLM\SOFTWARE\Policies\Google\Chrome\ExtensionSettings
값(REG_SZ): "<고정확장ID>"
데이터(JSON):
{
  "installation_mode": "force_installed",
  "update_url": "https://ext.corp.local/openclaude/update.xml",
  "toolbar_pin": "force_pinned"
}
```

전역 기본 차단 + 화이트리스트(보안 강화):
```json
{
  "*": { "installation_mode": "blocked" },
  "<고정확장ID>": {
    "installation_mode": "force_installed",
    "update_url": "https://ext.corp.local/openclaude/update.xml"
  }
}
```

**macOS (managed plist / MDM):** 도메인 `com.google.Chrome`(Edge: `com.microsoft.Edge`), 키 `ExtensionInstallForcelist`(배열) → `"<고정확장ID>;https://ext.corp.local/openclaude/update.xml"`. Jamf/Intune로 Configuration Profile 푸시.

**Linux:** `/etc/opt/chrome/policies/managed/openclaude.json`:
```json
{ "ExtensionInstallForcelist": ["<고정확장ID>;https://ext.corp.local/openclaude/update.xml"] }
```

#### 동반 정책

| 정책 | 목적 |
|---|---|
| `NativeMessagingAllowlist` = `["com.anthropic.open_claude_in_chrome"]` | 네이티브 호스트 통신 허용(Blocklist=`*` 환경 대비) |
| `ExtensionInstallSources` (필요 시) | 사내 .crx 출처 허용 |
| 사내 루트 CA 배포 | `ext.corp.local` HTTPS 신뢰 |

## 2. 네이티브 메시징 호스트 프로비저닝

### 2.1 배포 산출물 (Node 런타임 동봉)

PC에 Node가 없을 수 있으므로 **portable Node 동봉**. 설치 경로 표준화:
```
C:\Program Files\OpenClaude\           (Windows 표준 설치 경로)
├─ node\node.exe                       (동봉 portable Node 18+ LTS)
├─ host\
│   ├─ native-host.js
│   ├─ mcp-server.js
│   ├─ node_modules\                   (오프라인 사전 설치: @modelcontextprotocol/sdk, zod)
│   └─ native-host-wrapper.cmd         (Windows wrapper)
└─ VERSION

/opt/openclaude/                        (macOS/Linux)
```

> `node_modules`는 에어갭이라 `npm install` 불가 → **오프라인 미러에서 사전 설치 후 통째로 패키징**하거나 사내 npm 레지스트리 운영. (host의 유일 의존성: MCP SDK + zod — 상세는 `02-아키텍처-및-의존성-오프라인화.md`)

### 2.2 Windows 네이티브 호스트 등록 (install.sh 미지원 영역)

Windows는 manifest 경로를 **레지스트리**로 가리킴(파일 자동 탐색 아님).

**(A) wrapper (`native-host-wrapper.cmd`):**
```bat
@echo off
"C:\Program Files\OpenClaude\node\node.exe" "C:\Program Files\OpenClaude\host\native-host.js"
```

**(B) manifest (`com.anthropic.open_claude_in_chrome.json`)** — `C:\Program Files\OpenClaude\host\`에 배치:
```json
{
  "name": "com.anthropic.open_claude_in_chrome",
  "description": "Open Claude in Chrome Native Messaging Host",
  "path": "C:\\Program Files\\OpenClaude\\host\\native-host-wrapper.cmd",
  "type": "stdio",
  "allowed_origins": ["chrome-extension://<고정확장ID>/"]
}
```
> `key` 고정 덕분에 `allowed_origins`가 **단일 표준값**. install.sh가 인자로 ID를 받던 분기가 제거됨.

**(C) 레지스트리 등록 (HKLM = 전체 사용자):**

| 브라우저 | 레지스트리 키 |
|---|---|
| Chrome | `HKLM\SOFTWARE\Google\Chrome\NativeMessagingHosts\com.anthropic.open_claude_in_chrome` |
| Edge | `HKLM\SOFTWARE\Microsoft\Edge\NativeMessagingHosts\com.anthropic.open_claude_in_chrome` |
| Brave | `HKLM\SOFTWARE\BraveSoftware\Brave-Browser\NativeMessagingHosts\com.anthropic.open_claude_in_chrome` |

각 키의 **`(기본값)`(REG_SZ)** = manifest 절대 경로. PowerShell 자동화:
```powershell
$manifest = "C:\Program Files\OpenClaude\host\com.anthropic.open_claude_in_chrome.json"
$key = "HKLM:\SOFTWARE\Google\Chrome\NativeMessagingHosts\com.anthropic.open_claude_in_chrome"
New-Item -Path $key -Force | Out-Null
Set-ItemProperty -Path $key -Name "(default)" -Value $manifest
```

> **HKLM vs HKCU:** 멀티유저 공용 PC는 HKLM(관리자 권한, 1회 등록으로 전 사용자 적용). 사용자별 격리는 HKCU. HKLM manifest 경로/wrapper는 **모든 사용자 읽기/실행 권한** 필요.

### 2.3 배포 자동화 채널

| 채널 | 권장 패키징 |
|---|---|
| **Intune** (Win) | `.intunewin` (Win32 앱) — 파일 복사 + 레지스트리 등록 + 정책 적용 |
| **SCCM/MECM** | MSI 또는 PowerShell 배포 패키지 |
| **GPO** | 소프트웨어 설치(MSI) + 정책(ADMX) 동시 푸시 |
| **macOS (Jamf)** | `.pkg`(파일+manifest를 `/Library/.../NativeMessagingHosts/`) + Configuration Profile |
| **Linux (Ansible/MDM)** | 시스템 디렉터리 manifest + `/etc/.../policies/managed/` 정책 |

**시스템 레벨 manifest 경로**(전체 사용자):
- Chrome(mac): `/Library/Google/Chrome/NativeMessagingHosts/`
- Chrome(Linux): `/etc/opt/chrome/native-messaging-hosts/`
> 현재 `install.sh`는 `$HOME/Library/...`(사용자별)만 처리 → 대규모 배포는 시스템 경로 전환 권장.

**MSI/패키지 설치 시 install.sh가 하던 일 대체:**
1. 파일 + portable node + node_modules 복사
2. wrapper 생성(절대 경로 박힌 `.cmd`/`.sh`)
3. 네이티브 manifest 배치(고정 `allowed_origins`)
4. 레지스트리(Win) 또는 시스템 manifest 디렉터리(mac/Linux) 등록
5. Claude Code에 MCP 서버 등록(§3.3)

## 3. 내부 LLM 게이트웨이 운영

### 3.1 토폴로지

```
                       ┌──────────────────────────────┐
 사내 PC들 ──HTTPS──▶  │  L4/L7 LB (HAProxy / nginx)   │
 (Claude Code)         │   - TLS 종단(사내 CA)         │
                       │   - 헬스체크 /health          │
                       └───────────────┬──────────────┘
                                       │
                       ┌───────────────┴──────────────┐
                       │  LiteLLM Proxy (Active/Active) │  2+ 인스턴스
                       │   - Anthropic 호환 엔드포인트  │  (stateless, HA)
                       │   - 키/예산/라우팅/로깅       │
                       │   - Redis(공유 상태/속도제한) │
                       └───────────────┬──────────────┘
                                       │ OpenAI 호환
                       ┌───────────────┴──────────────┐
                       │  vLLM 서버 풀 (GPU 노드)       │
                       │   - 모델 A x N, 모델 B x M    │
                       │   - LiteLLM가 라운드로빈/가중  │
                       └──────────────────────────────┘
```

### 3.2 핵심 설계 포인트

| 항목 | 권장 |
|---|---|
| **Anthropic 호환** | Claude Code는 Anthropic Messages API 기대 → LiteLLM의 **`/v1/messages` unified 엔드포인트** 사용 |
| **HA** | LiteLLM stateless 다중 인스턴스 + Redis로 rate-limit/예산 공유. LB 뒤 배치 |
| **로드밸런싱** | LiteLLM `model_list`에 동일 모델 다중 vLLM 백엔드 등록 → least-busy/round-robin |
| **모델 호스팅** | vLLM은 가중치를 **로컬 디스크/사내 오브젝트 스토리지**에서 로드(허브 접근 불가) |
| **인증** | 사용자/팀별 LiteLLM virtual key 발급(감사/예산 관리) |
| **TLS** | 사내 CA 인증서, 모든 PC에 루트 CA 신뢰 사전 배포(확장 update 서버와 동일 CA 재사용) |

### 3.3 Claude Code 클라이언트 표준 설정 배포

**환경변수(시스템 전역):**
```bash
ANTHROPIC_BASE_URL=https://llm-gw.corp.local
ANTHROPIC_AUTH_TOKEN=<LiteLLM virtual key>
ANTHROPIC_MODEL=<사내모델명, 예: corp-llm-large>
ANTHROPIC_SMALL_FAST_MODEL=<경량모델명>
```

| OS | 배포 방법 |
|---|---|
| Windows | GPO "환경 변수" 정책 또는 설치 스크립트 `setx /M` (시스템 변수) |
| macOS | LaunchAgent/Configuration Profile, 또는 `/etc/profile.d/openclaude.sh` |
| Linux | `/etc/profile.d/openclaude.sh` |

**MCP 서버 표준 등록**(설치 스크립트에 포함):
```bash
claude mcp add open-claude-in-chrome -- \
  "C:\Program Files\OpenClaude\node\node.exe" \
  "C:\Program Files\OpenClaude\host\mcp-server.js"
```
> ⚠️ 위 `claude mcp add`는 **Claude Code 기준의 옛 절차**입니다. **본 환경은 opencode**이므로 MCP 등록은 `opencode.json`의 `mcp` 항목으로 대체됩니다([06 가이드](./06-셀프서비스-설치-가이드.md) §3, [05 문서](./05-확정-아키텍처-opencode-및-PoC.md) §2.2).

## 4. 멀티유저 / 멀티세션 운영

### 4.1 mcp-server.js의 멀티플렉싱은 "단일 PC 내" 공유 구조

- **PRIMARY/CLIENT 모드**: 첫 MCP 서버가 `127.0.0.1:18765`를 바인딩 → PRIMARY(네이티브 호스트 소켓 소유). 이후 서버는 `EADDRINUSE` → CLIENT로 PRIMARY에 TCP 접속.
- **분류 로직**: PRIMARY는 인바운드 소켓을 ~500ms 대기로 분류. `client_hello` 즉시 전송=클라이언트, 침묵=네이티브 호스트.
- **요청 라우팅**: 클라이언트 요청은 `c<clientId>_<id>` 프리픽스로 네이티브 호스트 전달, `clientRequestMap`으로 역라우팅.
- **단일 네이티브 호스트 제약**: `setupNativeHostConnection`에서 이미 호스트가 있으면 "Another browser profile is already connected" 거부 → **PC당 브라우저 프로필 1개**만 연결.

핵심: **포트 18765는 `127.0.0.1` 바인딩(루프백)** → 네트워크 미노출, **PC 간 포트 충돌 없음**. 각 PC가 독립적으로 자기 18765 사용.

### 4.2 동시성 매트릭스

| 시나리오 | 동작 | 운영 시사점 |
|---|---|---|
| **1 PC, 1 사용자, N Claude 세션** | PRIMARY 1 + CLIENT N-1, 한 브라우저 공유 | 정상 지원. N 세션이 동시에 같은 탭 조작 시 상호 간섭 가능 → 세션마다 `tabs_create_mcp`로 자기 탭 사용 권장 |
| **1 PC, 다중 OS 사용자(공용 PC)** | 포트는 PC 전역 | **주의**: 동시 로그인(RDP/멀티세션 Windows)이면 두 사용자가 같은 18765 경합 → 첫 사용자 PRIMARY 독점. **공용 멀티세션 PC 비권장**, 1인 1세션 PC 권장 |
| **다수 PC** | 각 PC 독립(루프백) | 충돌 없음. 각 PC가 자기 LiteLLM 키로 게이트웨이 호출 |

> 보안 관점에서 인증 부재로 인한 세션 간 격리 문제는 `03-보안-및-거버넌스.md` §2.2 참조.

### 4.3 충돌 회피 / 표준화 권장

| 권장 | 이유 |
|---|---|
| **PC당 단일 브라우저 프로필 표준** | 네이티브 호스트는 1개만 연결 허용 |
| **공용/멀티세션 OS 환경 회피** | 포트 18765 전역 경합. 불가피하면 사용자별 포트를 config.json `port`로 분리 + 사용자별 프로필 격리 |
| **포트 표준 유지(18765)** | 변경 시 mcp-server.js·native-host.js 양쪽 반영(config.json은 양측이 같은 파일을 읽어 안전) |
| **stale 서버 정리 런북화** | 재접속 시 죽은 PRIMARY가 포트를 쥔 경우 대비(§6) |

## 5. 업데이트 / 패치 관리 (에어갭)

### 5.1 4계층 버전 매트릭스

| 계층 | 산출물 | 전파 메커니즘 | 롤백 |
|---|---|---|---|
| 확장(MV3) | `.crx` + `update.xml` | 내부 update 서버 버전 범프 → 5시간 주기 자동 폴링 | `update.xml`을 이전 .crx로 |
| 네이티브 호스트 + Node | MSI/pkg | Intune/SCCM/Jamf 재배포 | 이전 MSI 재배포 |
| MCP 서버 | host/ 파일(MSI 동봉) | 호스트와 동일 패키지 | 〃 |
| Claude Code CLI | 사내 npm(Verdaccio) tarball | `npm i -g` from 사내 레지스트리 | 버전 핀 고정 |
| LLM 모델/게이트웨이 | vLLM/LiteLLM 컨테이너·가중치 | 사내 레지스트리 이미지 롤아웃 | 이전 태그 재배포 |

### 5.2 스테이징 → 검증 → 링 배포 파이프라인

```
[오프라인 빌드] → [스테이징 링(IT/QA 5~10 PC)] → [파일럿 링(1개 팀)] → [전사 배포]
   서명/패키징         스모크+회귀 검증              실사용 1주 관찰        링별 정책 타겟팅
```

- **스모크 테스트(필수)**: *"Navigate to reddit.com and take a screenshot"*(reddit이 원본 블록리스트라 성공=언블록 동작 증명) + 사내 LLM 응답 확인.
- **링 배포**: GPO/Intune 그룹별로 update.xml/MSI 타겟 단계 노출.
- **버전 호환 검증**: 확장 ↔ 네이티브 호스트 ↔ MCP 서버는 **와이어 프로토콜**(newline JSON, `tool_request`/`tool_response`)이 계약. 프로토콜 변경 시 세 계층 동시 배포 필요 → **세 계층을 같은 MSI/패키지에 묶어** 버전 스큐 방지(확장만 별도 update 서버라 특히 주의).

### 5.3 롤백 절차(요약)
1. 확장: `update.xml`을 직전 버전 .crx로 교체 → 강제 폴링(`chrome://policy` → Reload policies, 또는 재시작)
2. 호스트/Node/MCP: Intune/SCCM 이전 버전 MSI 재할당
3. 게이트웨이/모델: LB에서 이전 LiteLLM/vLLM 태그로 전환

## 6. 모니터링 / 지원 / 장애 런북

### 6.1 헬스체크 / 로그 수집

| 대상 | 로그 위치 | 수집 방법 |
|---|---|---|
| 확장 서비스 워커 | `chrome://extensions` → "Inspect views: service worker" | 원격 수집 어려움 → 확장 내 로깅 후 사내 엔드포인트 전송(폐쇄망 내부) |
| MCP 서버 | **stderr** → Claude Code MCP 로그(`~/.claude/logs` 등) | 로그 수집 에이전트로 디렉터리 tail |
| 네이티브 호스트 | wrapper에서 stderr를 파일로 리다이렉트 권장 | 〃 |
| LiteLLM | proxy access/spend 로그 | 중앙 로그(폐쇄망 ELK/Loki) |
| vLLM | 서버 로그/메트릭 | Prometheus(내부) |

> **개선 제안**: native-host-wrapper에 로그 리다이렉트 추가 →
> Windows: `... native-host.js >> "%PROGRAMDATA%\OpenClaude\nh.log" 2>&1`
> mac/Linux: `exec node native-host.js >> /var/log/openclaude/nh.log 2>&1`

### 6.2 장애 런북

| 증상 | 원인 | 조치 |
|---|---|---|
| **"Browser extension is not connected"** | MCP 서버는 떴으나 네이티브 호스트 미연결 | ① 웹페이지 열어 서비스 워커 깨우기 ② wrapper 존재/실행권한 ③ 레지스트리/manifest 경로 검증 ④ 브라우저 완전 재시작(전 창 종료) |
| **포트 18765 점유 / 재접속 후 툴 즉시 실패** | stale PRIMARY가 포트 보유 | `pkill -f "node.*mcp-server"`(mac/Linux) / `Get-Process node | Stop-Process`(Win, 주의) 후 `/mcp` 재연결 |
| **"Another browser profile is already connected"** | 두 번째 브라우저 프로필이 연결 시도 | PC당 단일 프로필 표준. 의도 시 한쪽 종료 |
| **확장 자동 비활성화/사라짐** | 개발자 모드 배포 또는 정책 누락 | `ExtensionInstallForcelist` 적용 확인(`chrome://policy`), `.crx` 강제설치 전환 |
| **확장 업데이트 안 됨** | update 서버 도달 불가/CA 미신뢰 | `ext.corp.local` HTTPS 신뢰, `update.xml` MIME/버전 확인, `chrome://extensions` Update |
| **LLM 응답 없음/인증 실패** | 게이트웨이 다운/키 만료/BASE_URL 오설정 | LB·LiteLLM `/health`, virtual key, `ANTHROPIC_BASE_URL` 검증 |
| **Retina/좌표 어긋남(macOS)** | `deviceScaleFactor` 미고정 | 확장 코드 불변식(background.js `deviceScaleFactor:1`) 회귀 여부 점검 |

### 6.3 정기 헬스 점검(자동화 권장)
- 합성 모니터: 대표 PC에서 주기적 스모크(reddit 스크린샷 + LLM ping) → 중앙 보고
- update 서버/LB/LiteLLM/vLLM: `/health` Prometheus blackbox
- 버전 드리프트 점검: 각 PC의 확장/호스트 MSI/Claude Code 버전 인벤토리 수집(Intune/SCCM)

## 핵심 메시지 3가지

1. **본 환경은 개발자 모드 허용 → 셀프서비스(Path A)가 기본.** 각 사용자가 Load unpacked + 자기 ID로 install 실행. `install.sh`가 ID를 인자로 받으므로 PC별 ID 상이 무방. **`key` 고정·`.crx`·강제설치는 중앙 통제·자동 업데이트가 필요할 때의 옵션(Path B).**
2. **단, Windows 지원(`install.ps1`·wrapper.bat·레지스트리)과 vendored node_modules·Node 반입은 배포 방식과 무관하게 필요**(현 install.sh는 mac/Linux 전용). 보안 하드닝(`03`)도 공통 필수.
3. **포트 18765는 PC-로컬(루프백)**이라 PC 간 충돌은 없지만, **PC당 단일 브라우저 프로필**·**1인 1세션 PC**를 표준으로 하고, 공용 멀티세션 OS는 포트 경합 때문에 피하라.
