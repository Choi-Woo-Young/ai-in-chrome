## 2. 종합 결론 (한 문단)

**이 프로젝트는 에어갭 "이식" 자체는 이례적으로 쉽고(빌드 스텝 0, 런타임이 100% 로컬 루프백, 의존성은 순수 JS 91개·실행은 SDK 1개), 진짜 작업량은 "이식"이 아니라 "재무장(re-arming)"에 있습니다.** 즉, ①LLM 두뇌를 오픈소스로 갈아끼우는 게이트웨이 구성과 ②이 도구가 의도적으로 제거한 안전장치(도메인 블록리스트·계획 승인·로컬 인증·감사)를 폐쇄망 엔터프라이즈 기준으로 **되살리는 것**, 그리고 ③Windows 지원 보완·배포(네이티브 호스트 등록)가 핵심입니다. **배포는 개발자 모드가 허용되므로 셀프서비스(각 사용자가 Load unpacked로 직접 추가)가 기본 경로**이며, `key` 고정·강제설치는 중앙 통제가 필요할 때의 옵션입니다. 순수 이식만으로 "동작"은 하지만, 다수 사용자 폐쇄망에서 **안전하게 운영**하려면 보안 High 항목이 도입 전제 조건입니다.

## 3. 핵심 교차 발견 (영역 간 종합)

### 3.1 중심 긴장: "가드레일 제거" 도구를 "가드레일 필수" 환경에 넣는다
원본 "Open Claude in Chrome"의 정체성은 **공식 확장의 제약(58개 도메인 블록리스트, 계획 승인, 권한 게이트)을 걷어낸 것**입니다(README, CLAUDE.md). 그런데 사내 폐쇄망 엔터프라이즈는 정반대로 **접근통제·승인·감사**를 요구합니다. → 커스터마이징의 본질은 "기능 추가"보다 **제거된 안전장치의 선별적 복원**입니다. (③ 전반)

### 3.2 영역 간 상충 1건과 해소
- **상충**: ②아키텍처 보고서는 도메인 화이트리스트·감사 로깅을 **"선택(optional)"** 으로, ③보안 보고서는 **"도입 전제 조건(High)"** 으로 분류.
- **해소(종합 판정)**: **다수 사용자 폐쇄망에서는 "필수"로 격상**한다. 단일 사용자 PoC 단계에서는 후순위 가능. → 본 로드맵에서는 **파일럿(전사 전) 단계의 게이트 통과 조건**으로 둔다.

### 3.3 비전(스크린샷) 필요성이 모델 선택과 코드 변경을 동시에 좌우
- `computer:screenshot`/`zoom`이 JPEG 이미지를 반환(①, ②) → **비전 모델을 쓰면** 공식 경험과 동등, **텍스트 모델만 쓰면** `read_page`/`find`/`ref` 기반 "텍스트-퍼스트"로 우회해야 함(①§4.2).
- 텍스트-퍼스트로 가면 ②의 코드 변경(`computer` 이미지 반환 → 텍스트 대체, `background.js:470~`)이 따라옴. → **모델 결정이 코드 변경 범위를 정함.** PoC에서 양쪽을 실측 비교 권장.

### 3.4 배포 방식: 개발자 모드 허용 → 셀프서비스가 기본
- **본 환경은 개발자 모드 활성화가 가능**하므로, 각 사용자가 직접 Load unpacked로 확장을 추가하는 **셀프서비스(Path A)가 기본 경로**. `install.sh`가 확장 ID를 인자로 받아 PC별 ID 상이도 무방(원본이 셀프서비스용으로 설계됨).
- 확장 `key` 고정·`.crx`·`ExtensionInstallForcelist` 강제배포(Path B)는 **중앙 통제·자동 업데이트·비기술 사용자 확대·무결성 중앙 보증**이 필요할 때의 **옵션**. (개발자 모드가 정책으로 차단될 때만 사용자 수 무관 필수)
- **단, Windows 지원(install.ps1·레지스트리)·vendored node_modules·보안 하드닝은 배포 방식과 무관하게 공통 필요.**

### 3.5 코드 수정 없이 끝나는 것 vs 반드시 코드를 건드려야 하는 것
| 코드 수정 불필요(설정/배포만) | 코드 수정 필요 |
|---|---|
| 내부 LLM 연결(`ANTHROPIC_BASE_URL` 등 환경변수) | 도메인 화이트리스트 + 감사 로깅(`navigate`/`callTool`/`handleToolRequest`) |
| 포트 변경(`config.json`) | 로컬 핸드셰이크 인증(`mcp-server.js` client_hello 검증) |
| MCP 등록(`.mcp.json`) | `update_plan` 자동승인 제거·쓰기성 도구 승인 게이트 |
| node_modules·Node·브라우저 오프라인 반입 | 비전 미지원 시 이미지→텍스트 대체 |
| 확장 강제설치 정책(GPO/레지스트리) | 확장 ID 고정(manifest `key` 추가) |

## 5. 단계별 도입 로드맵 (실현 가능 순서)

```
[Phase 0] 준비·반입        [Phase 1] PoC(1대)        [Phase 2] 보안 하드닝       [Phase 3] 파일럿(1팀)      [Phase 4] 전사 배포
 오프라인 아티팩트          단일 PC 기능 검증           High 보안항목 코드화        강제배포·감사·운영검증       링 배포·모니터링·런북
 (의존성/Node/모델)         (텍스트-퍼스트 18툴)        (인증/화이트/승인/감사)      (5~10PC→1팀)               (전 PC)
```

### Phase 0 — 준비 및 반입 (코드 변경 없음)
- 외부 PC에서 `cd host && npm ci` → `node_modules` 포함 `.tgz` (아 §2.2)
- Node v20 LTS + 사내 표준 Chromium + Claude Code 오프라인 패키지 반입
- 모델 가중치(`.safetensors`) 무결성 검증 후 반입, 내부 모델 스토리지 적재 (엘 §5)
- **반입 게이트**: 확장/호스트/SDK/가중치 SHA-256 + 서명 대조 (보 §5)

### Phase 1 — PoC (단일 PC, 기능 검증)
- LiteLLM + vLLM(Qwen3-Coder) 단일 인스턴스 + `ANTHROPIC_BASE_URL` 연결 (엘 §1)
- 개발자 모드(Load unpacked)로 1대 로딩 → 18툴 중 **실동작 12툴** 검증
- **텍스트-퍼스트**(`read_page`/`find`/`ref`)로 핵심 플로우 동작 확인 (엘 §4.2)
- 툴콜 파서 정상성·LiteLLM 블록순서·count_tokens 오차 실측 (엘 §7 ⚠️검증필요)
- **비전 분기 결정**: VL 모델 추가 시 스크린샷 워크플로 vs 텍스트-퍼스트 비교

### Phase 2 — 보안 하드닝 (High 항목 코드화) ← **전사 전 필수 게이트**
- 로컬 핸드셰이크 인증, 도메인 화이트리스트, `update_plan` 승인 게이트, `javascript_tool` 통제, 감사 로깅 (보 §6 High 1~5)
- 변경 후 PoC 재검증(회귀)

### Phase 3 — 파일럿 (5~10 PC → 1개 팀)
- 셀프서비스 배포 아티팩트 묶음 + 사용자 온보딩 가이드 (개발자 모드 허용 → Path A 기본)
- Windows install.ps1·wrapper.bat·레지스트리 등록·Node 반입 (배 §2)
- (옵션 Path B) 중앙 통제 필요 시 `key` 고정·`.crx`·강제설치 정책 (배 §1)
- LiteLLM HA + virtual key, 합성 모니터·감사 로그 중앙 수집 (배 §3·§6)
- 실사용 1주 관찰, 온보딩 체크리스트 검증 (배 §7)

### Phase 4 — 전사 배포
- 링(ring) 단계 노출, 4계층 버전 관리·롤백 파이프라인 (배 §5)
- 운영 런북·헬스체크 정착 (배 §6)

## 6. LLM 모델 최종 권고 (요약 — 상세는 01 문서)

| 용도 | 권장 | 근거 |
|---|---|---|
| **메인(1순위)** | **Qwen3-Coder 30B-A3B (FP8)** | Apache-2.0(법무 안전), MoE 활성 3B로 H100 1~2장 고동시성, 256K 컨텍스트, 전용 툴콜 파서 |
| **최고성능(옵션)** | **DeepSeek-V3 (FP8)** | 에이전틱 최상위권, 단 H100×8 1노드 점유 → 고난도 작업만 라우팅 |
| **경량(SMALL_FAST)** | 소형 Qwen3(예 8B FP8) | 요약·제목 등 보조 작업 |
| **비전(조건부)** | **Qwen2.5-VL 72B (AWQ)** | 스크린샷 기반 워크플로 필요 시. 비전 vs 텍스트-퍼스트는 PoC 실측 후 결정 |

- 게이트웨이: **LiteLLM `/v1/messages` unified + vLLM** (에어갭에서 Bedrock/Vertex 불가하므로 유일 성립 경로)
- ⚠️ 반입 전 재검증: 최신 모델 세대 가용성, vLLM 버전별 툴콜 파서, LiteLLM 안전 버전(악성 1.82.7/1.82.8 차단), 라이선스 조항 (엘 §7)

## 10. 가장 먼저 할 일 (Top 3 Quick Start)

1. **Phase 0 반입 패키지 만들기** — 외부 PC에서 `cd host && npm ci` 후 `node_modules`+Node+모델을 무결성 검증해 반입. (코드 변경 0, 가장 빠른 착수)
2. **PoC 1대 + Qwen3-Coder + LiteLLM 연결** — `ANTHROPIC_BASE_URL`만으로 동작 확인하고 **오픈모델 툴콜 안정성(최대 리스크)** 을 조기 판별.
3. **셀프서비스 배포 아티팩트 준비** — 개발자 모드 허용 환경이므로 vendored node_modules·Node·install 스크립트(Windows install.ps1 포함)·사용자 가이드를 묶어, 각 사용자가 직접 Load unpacked로 추가하게 한다. (`key` 고정·강제배포는 중앙 통제 필요 시 옵션 Path B)

> 모든 상세 절차·코드 before/after·설정 예시는 01~04 개별 문서를 참조하십시오.
