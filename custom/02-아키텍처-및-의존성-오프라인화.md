# 에어갭(폐쇄망) 이식 검토 보고서: 아키텍처 & 의존성 오프라인화

> 하네스팀 검토 ② — 아키텍처 & 의존성 오프라인화
> 전제: 완전 에어갭 · 고사양 GPU 자체호스팅 · 사내 표준 OS Windows 가능성

> ⚠️ **아키텍처 업데이트 ([05 문서](./05-확정-아키텍처-opencode-및-PoC.md) 우선):** 두뇌가 **Claude Code → opencode**로 확정. §3의 "Claude Code ↔ 내부 LLM 연결(`ANTHROPIC_BASE_URL`)" 부분은 **opencode 설정(OpenAI 직결)** 으로 대체됩니다(05 문서 §2). 그 외 **의존성 오프라인화·Windows 지원·코드 변경 지점(화이트리스트/감사/비전 대체)·빌드 단순성 substance는 그대로 유효**하며, 배포는 **기존 VDI에 셀프서비스 1회 설치**(05 §5 / 06 가이드)가 본 환경 확정 경로입니다.

## 0. 핵심 요약 (Executive Summary)

이 프로젝트는 에어갭 이식에 **매우 유리한 구조**입니다. 결정적 이유는 세 가지입니다.

1. **빌드 스텝이 전혀 없음** — `extension/`, `host/` 모두 plain JS(번들러/트랜스파일러 불필요). `CLAUDE.md:9` 명시: *"There is no build step."*
2. **런타임 외부 의존성이 사실상 없음** — 모든 통신이 `127.0.0.1` 로컬 루프백(TCP `:18765`) + Chrome 네이티브 메시징(stdin/stdout)으로만 이루어짐. 인터넷으로 나가는 코드 경로가 **없습니다**.
3. **유일한 npm 의존성은 `@modelcontextprotocol/sdk` 1개** (`host/package.json:8`) — 단, 전이 의존성까지 포함하면 91개 패키지 트리.

외부망 의존은 **설치 시점(install-time)에만** 존재하며, 런타임에는 없습니다. 따라서 에어갭 이식의 본질은 "코드 수정"보다 **"npm 의존성과 Node 런타임의 오프라인 반입 + Windows 등록 절차 + 내부 LLM 게이트웨이 설정"** 세 가지로 압축됩니다.

### 1.1 오프라인 동작 검증 (코드 근거)

- 네이티브 호스트 연결: `connectNativeHost()` → `chrome.runtime.connectNative(NATIVE_HOST_NAME)` (`background.js:29-32`), 호스트명 `com.anthropic.open_claude_in_chrome` (`background.js:9`). **로컬 프로세스만 기동** — 외부망 불필요.
- 브리지는 `127.0.0.1:18765` 명시적 루프백 바인딩: `tcpServer.listen(TCP_PORT, "127.0.0.1", ...)` (`mcp-server.js:405`), `tcpSocket.connect(TCP_PORT, "127.0.0.1", ...)` (`native-host.js:71`). **외부 인터페이스에 바인딩하지 않음**.
- 확장은 `chrome://extensions` Load unpacked로 로딩(스토어 불필요, `README.md:95-98`) → 폐쇄망 OK.

## 2. npm/Node 의존성 오프라인화 방안

### 2.1 vendoring(node_modules 사전 번들) vs 사내 프라이빗 레지스트리 비교

| 항목 | A. node_modules 사전 번들링 (vendoring) | B. 사내 레지스트리 미러 (Verdaccio/Nexus/Artifactory) |
|---|---|---|
| 적합도 | **이 프로젝트에 최적** | 과한 인프라 |
| 작업량 | 외부 PC 1회 `npm ci` → 폴더 통째로 반입 | 레지스트리 구축 + 91개 패키지 publish/proxy |
| 재현성 | `package-lock.json` 존재로 완벽 고정 | 동일 |
| 다수 PC 배포 | 폴더 복사만으로 끝 (네이티브 빌드 없음) | 각 PC에 `.npmrc` registry 설정 필요 |
| 권장도 | ★★★ **1순위 권장** | 사내에 이미 Nexus가 있고 다른 프로젝트도 많을 때만 |

**근거**: 의존성 91개가 **전부 순수 JS**(express, zod, jose 등 — 네이티브 애드온/C++ 컴파일 없음). OS/아키텍처 독립적이라 한 번 받은 `node_modules`를 Windows·macOS·Linux에 그대로 복사해도 동작. 빌드 스텝도 없으므로 vendoring이 압도적으로 단순.

### 2.2 `@modelcontextprotocol/sdk` 및 하위 의존성 반입 절차 (vendoring)

**외부망 연결된 PC에서 (1회):**
```bash
cd host
npm ci          # package-lock.json 기준 정확히 재현 설치 (91개 패키지)
```

**반입 패키징:**
```bash
cd host
tar czf host-node_modules.tgz node_modules package.json package-lock.json
```

**폐쇄망 각 PC에서:**
```bash
cd <project>/host
tar xzf host-node_modules.tgz   # node_modules 복원, npm install 불필요
```
- `.gitignore:1`이 `node_modules/`를 제외하므로 git이 아닌 별도 아티팩트로 배포.

**대안(레지스트리 방식):**
```bash
# 폐쇄망 PC의 host/.npmrc
registry=https://nexus.사내도메인/repository/npm-group/
npm ci
```

### 2.3 Node.js 런타임 오프라인 배포

- `install.sh:36`은 `command -v node`로 존재만 확인(설치 안 함) → **Node 런타임은 별도 반입 필수**.
- 권장: 공식 Node.js v20 LTS 오프라인 설치본을 사내 표준으로 배포.
  - **Windows**: `node-v20.x.x-x64.msi` 또는 무설치 zip → SCCM/Intune 배포.
  - **macOS/Linux**: `.pkg` / `.tar.xz` 또는 사내 패키지 미러.
- v18+ 요구(`README.md:81`). v20 LTS 권장.

## 3. Claude Code ↔ 내부 LLM 게이트웨이 연결 설정

이 부분은 **이 레포의 코드 변경 없이 환경변수/설정만으로 가능**합니다. 이 프로젝트는 MCP 서버일 뿐 LLM을 직접 호출하지 않으며, LLM 호출은 Claude Code 본체가 담당합니다.

### 3.1 내부 게이트웨이(LiteLLM 등) 지정

```bash
# 내부 LLM 게이트웨이 (Anthropic API 호환 프록시; LiteLLM이 변환 담당)
export ANTHROPIC_BASE_URL="https://llm-gw.사내도메인/anthropic"
export ANTHROPIC_AUTH_TOKEN="<사내 발급 토큰>"      # 또는 ANTHROPIC_API_KEY
export ANTHROPIC_MODEL="사내-claude-호환-모델명"
# 텔레메트리/자동 업데이트 차단(폐쇄망 외부 호출 시도 방지)
export DISABLE_TELEMETRY=1
export DISABLE_AUTOUPDATER=1
```
> LiteLLM을 게이트웨이로 두면 Claude Code는 변경 없이 `ANTHROPIC_BASE_URL`만 바꾸면 됨. (상세는 `01-LLM-모델-및-서빙-인프라.md` 참조)

### 3.2 MCP 등록 절차의 에어갭/다수 PC 표준화

기존(온라인) 절차(`README.md:119`, `install.sh:129`):
```bash
claude mcp add open-claude-in-chrome -- node /absolute/path/to/host/mcp-server.js
```

다수 PC 표준화를 위해 **프로젝트 단위 `.mcp.json`**(레포에 체크인) 권장:
```jsonc
// 프로젝트 루트 .mcp.json
{
  "mcpServers": {
    "open-claude-in-chrome": {
      "command": "node",
      "args": ["./host/mcp-server.js"]
    }
  }
}
```
또는 사내 표준 설치 경로(예: `C:\Tools\open-claude-in-chrome`)로 고정하고 절대 경로를 박아 배포.

## 4. 코드 커스터마이징 필요 지점 (파일/라인 단위)

> ⚠️ 본 절의 화이트리스트/감사 로깅은 **보안 검토(`03-보안-및-거버넌스.md`)에서 도입 전제 조건(High)으로 분류**됩니다. 단순 선택이 아니라 폐쇄망 멀티유저 도입 시 필수에 가깝습니다.

### 4.1 내부 화이트리스트 / 감사 로깅 추가

이 프로젝트의 강점은 **도메인 블록리스트가 없다는 점**(`CLAUDE.md:7`, `README.md:42`). 그러나 폐쇄망 엔터프라이즈에서는 반대로 **"허용된 내부 도메인만 접근 + 감사 로깅"** 요구가 생깁니다. 적용 위치는 `navigate` 핸들러.

**위치**: `extension/background.js:405` `async navigate(args)` 내부, URL 정규화 직후(`:426` `chrome.tabs.update` 직전).

**Before** (`background.js:421-427`):
```js
      try {
        new URL(targetUrl); // Validate URL before passing to Chrome
      } catch {
        return { content: [{ type: "text", text: `Invalid URL: "${url}". Could not parse as a valid URL.` }] };
      }
      await chrome.tabs.update(tabId, { url: targetUrl });
```

**After** (화이트리스트 + 감사 로깅):
```js
      let parsed;
      try {
        parsed = new URL(targetUrl);
      } catch {
        return { content: [{ type: "text", text: `Invalid URL: "${url}". Could not parse as a valid URL.` }] };
      }

      // === 사내 폐쇄망 정책: 내부 도메인 화이트리스트 ===
      const ALLOWED_HOSTS = [
        "intranet.사내도메인",
        "wiki.사내도메인",
        "jira.사내도메인",
        // 필요한 내부 호스트만 추가
      ];
      const hostOk = ALLOWED_HOSTS.some(
        (h) => parsed.hostname === h || parsed.hostname.endsWith("." + h)
      );
      if (!hostOk) {
        console.warn(`[AUDIT][BLOCKED] navigate ${parsed.href} tab=${tabId} @${new Date().toISOString()}`);
        return { content: [{ type: "text", text: `차단됨: "${parsed.hostname}" 은(는) 사내 허용 도메인이 아닙니다.` }] };
      }
      console.info(`[AUDIT][NAVIGATE] ${parsed.href} tab=${tabId} @${new Date().toISOString()}`);

      await chrome.tabs.update(tabId, { url: targetUrl });
```
> **참고**: `javascript_tool`(`background.js:727`)은 임의 JS 실행이 가능하므로 화이트리스트를 우회할 수 있습니다. 강한 격리가 필요하면 이 도구도 정책 검토 대상(보안 보고서 §1.2, §3.2 참조).

코드 변경 없이 포트만 바꾸려면 `~/.config/open-claude-in-chrome/config.json`(`mcp-server.js:24`, `native-host.js:16`):

### 4.4 내부 LLM이 비전(Vision) 미지원일 경우: 이미지 → 텍스트 대체

자체 호스팅 LLM이 이미지 입력을 지원하지 않으면, 스크린샷 반환 도구가 무의미·오류 유발. 영향 위치:

| 도구/동작 | 파일:라인 | 이미지 반환 부분 |
|---|---|---|
| `computer` → `screenshot` | `background.js:470-486` | `{ type: "image", data: base64, mimeType: "image/jpeg" }` (`:483`) |

## 5. Windows 지원 (install.sh가 macOS/Linux 전용)

`install.sh:99-121`은 `uname` 분기에서 `Darwin`/`Linux`만 처리하고, Windows는 `*)` 분기에서 종료(`install.sh:116-120`). Windows 네이티브 메시징은 **레지스트리 등록**과 **`.bat` 래퍼**가 필요.

### 5.1 Windows 네이티브 메시징 동작 방식

- Windows의 Chrome/Edge는 네이티브 호스트를 **레지스트리 키**로 찾음:
  - Chrome: `HKCU\Software\Google\Chrome\NativeMessagingHosts\<host-name>`
  - Edge: `HKCU\Software\Microsoft\Edge\NativeMessagingHosts\<host-name>`
  - 키의 (기본값) = 매니페스트 JSON 파일의 **절대 경로**.
- 매니페스트의 `path`는 **`.bat`(또는 `.exe`) 래퍼**를 가리켜야 함(Windows는 `.js`를 직접 실행 못함).

### 5.2 파일 1: `native-host-wrapper.bat`

```bat
@echo off
REM host\native-host-wrapper.bat — Chrome가 네이티브 메시징으로 호출
node "%~dp0native-host.js"
```
- `%~dp0` = 이 .bat이 있는 디렉터리(=`host\`). node가 PATH에 없으면 절대경로(예: `"C:\Program Files\nodejs\node.exe"`)로 박음.

### 5.3 파일 2: 매니페스트 (Windows)

```json
{
  "name": "com.anthropic.open_claude_in_chrome",
  "description": "Open Claude in Chrome Native Messaging Host",
  "path": "C:\\Tools\\open-claude-in-chrome\\host\\native-host-wrapper.bat",
  "type": "stdio",
  "allowed_origins": [ "chrome-extension://<EXTENSION_ID>/" ]
}
```

### 5.4 파일 3: `install.ps1` (PowerShell 설치 스크립트)

```powershell
# install.ps1  — Usage: .\install.ps1 -ExtensionIds "<id1>","<id2>"
param([Parameter(Mandatory=$true)][string[]]$ExtensionIds)

$ErrorActionPreference = "Stop"
$ScriptDir = Split-Path -Parent $MyInvocation.MyCommand.Path
$HostDir   = Join-Path $ScriptDir "host"
$HostName  = "com.anthropic.open_claude_in_chrome"
$Wrapper   = Join-Path $HostDir "native-host-wrapper.bat"

# 0) node 확인
if (-not (Get-Command node -ErrorAction SilentlyContinue)) { throw "node not found in PATH" }

# 1) 의존성(에어갭: node_modules가 이미 반입되어 있어야 함)
if (-not (Test-Path (Join-Path $HostDir "node_modules"))) {
  throw "host\node_modules 없음. 에어갭 반입 아티팩트를 먼저 복원하세요(§2.2)."
}

# 2) .bat 래퍼 생성
@"
@echo off
node "%~dp0native-host.js"
"@ | Set-Content -Encoding ASCII $Wrapper

# 3) allowed_origins 구성
$origins = ($ExtensionIds | ForEach-Object { "    `"chrome-extension://$_/`"" }) -join ",`n"
$manifest = @"
{
  "name": "$HostName",
  "description": "Open Claude in Chrome Native Messaging Host",
  "path": "$($Wrapper -replace '\\','\\')",
  "type": "stdio",
  "allowed_origins": [
$origins
  ]
}
"@
$manifestPath = Join-Path $HostDir "$HostName.json"
$manifest | Set-Content -Encoding UTF8 $manifestPath

# 4) 레지스트리 등록 (Chrome + Edge + Brave, 사용자 단위 HKCU)
$targets = @{
  "Chrome" = "HKCU:\Software\Google\Chrome\NativeMessagingHosts\$HostName";
  "Edge"   = "HKCU:\Software\Microsoft\Edge\NativeMessagingHosts\$HostName";
  "Brave"  = "HKCU:\Software\BraveSoftware\Brave-Browser\NativeMessagingHosts\$HostName";
}
foreach ($name in $targets.Keys) {
  $key = $targets[$name]
  New-Item -Path $key -Force | Out-Null
  Set-ItemProperty -Path $key -Name "(default)" -Value $manifestPath
  Write-Host "Registered for $name -> $key"
}

Write-Host "`nDone. Next:"
Write-Host "  1. 브라우저 완전 재시작(모든 창 닫기)"
Write-Host "  2. claude mcp add open-claude-in-chrome -- node `"$HostDir\mcp-server.js`""
```

- JSON 내 Windows 경로는 백슬래시를 `\\`로 이스케이프.
- HKCU(사용자 단위) 등록은 관리자 권한 불필요. 전사 배포 시 HKLM + GPO도 가능(배포 보고서 §2.2 참조).

> 대규모 배포에서는 위 PowerShell을 그대로 쓰기보다 **배포 보고서(`04-배포-및-운영.md`)의 HKLM·고정 확장 ID(manifest `key`) 방식**을 권장합니다.

## 6. 빌드/배포 단순성 — 에어갭 이식에 유리한 점

1. **빌드 스텝 0개** (`CLAUDE.md:9`): 트랜스파일러/번들러·소스맵·minify 단계 전무 → 빌드 툴체인 반입 불필요.
2. **런타임 의존성 0개 (브라우저 측)**: `extension/`은 npm 패키지를 전혀 import하지 않음(전부 Chrome 확장 API + plain JS). `host/`만 SDK 의존.
3. **순수 JS 의존성 트리** (네이티브 빌드 없음): 91개 전이 의존성이 모두 순수 JS → `node-gyp`/Python/C++ 컴파일러 불필요 → OS·아키텍처 무관 폴더 복사 배포.
4. **lockfile 존재** (`host/package-lock.json`): `npm ci` 1회로 정확히 재현, 버전 드리프트 없음.
5. **전 통신 로컬 루프백** (`mcp-server.js:405`, `native-host.js:71`): 방화벽/프록시 무관, 외부 인터페이스 노출 없음.
6. **확장 배포가 스토어 불필요**: Load unpacked(`README.md:95-98`) → Web Store 접근 불필요.

**결론**: 에어갭 이식의 전체 작업은 사실상 **(a) `node_modules` + Node 런타임 + 브라우저 오프라인 반입(§2), (b) Windows 등록 스크립트 작성(§5), (c) 내부 LLM 게이트웨이 환경변수 설정(§3)** 세 가지로 끝나며, **소스 코드 수정은 보안 강화(화이트리스트/감사 로깅)·비전 비활성·리네이밍에 한정**됩니다.
