## 1. 확정 아키텍처

```
[폐쇄망 — 각 사용자 VDI]
 opencode (터미널 에이전트 = 두뇌, MCP 클라이언트)
    ├─ OpenAI API ─▶ vLLM (중앙 GPU 서버)
    │                 ├─ gpt-oss            (텍스트 에이전트 후보)
    │                 ├─ Qwen3.5-27B        (텍스트 에이전트 후보)
    │                 └─ Qwen2.5-VL 등       (비전 — 스크린샷 처리)
    └─ MCP ─▶ mcp-server.js ─▶ native-host.js ─▶ 확장 ─▶ 브라우저   ← 이 프로젝트 재사용(손발)
```

- **두뇌 = opencode**: LLM이 판단 → MCP 도구 호출 → 결과 → 재판단하는 에이전트 루프 주체.
- **손발 = 확장 + host**: 18개 도구(navigate·read_page·find·computer·form_input·js 등) 실행.
- **LLM = 중앙 vLLM**: VDI가 아니라 중앙 GPU 서버. opencode가 OpenAI API로 호출.

> **LiteLLM 불필요(기본):** opencode가 OpenAI API를 쓰고 vLLM이 OpenAI API를 네이티브 제공하므로 변환 레이어 불필요. 다중 vLLM 인스턴스 로드밸런싱·가상키·중앙 로깅이 필요할 때만 **옵션**으로 LiteLLM을 앞단에 둠.

## 2. opencode 에어갭 구성

opencode는 오픈소스(MIT)·터미널 코딩 에이전트로, **커스텀 OpenAI 호환 프로바이더**와 **MCP 서버**를 지원합니다. 폐쇄망에서는 아래 3가지를 반드시 처리합니다.

### 2.1 에어갭 필수 처리 (반입 전 체크)

| 처리 | 이유 |
|---|---|
| **커스텀 프로바이더 수동 정의** | opencode 기본은 모델 카탈로그를 `models.dev`에서 조회 → 폐쇄망 불가. 자체 vLLM을 OpenAI 호환 프로바이더로 직접 등록해 외부 조회 제거 |
| **자동 업데이트·텔레메트리 차단** | 외부 호출 시도 방지 |
| **바이너리 vendoring** | 설치 스크립트/npm 대신 사전 빌드 바이너리(또는 아티팩트)를 반입 번들에 포함(06 §1) |

### 2.2 설정 예시 (`opencode.json` — 개념, 버전 문서로 키 확인)

```jsonc
{
  // 자체 vLLM을 OpenAI 호환 프로바이더로 등록 (models.dev 외부 조회 불필요)
  "provider": {
    "local-vllm": {
      "npm": "@ai-sdk/openai-compatible",
      "options": { "baseURL": "http://vllm.internal:8000/v1", "apiKey": "dummy" },
      "models": {
        "gpt-oss":      { "name": "gpt-oss" },
        "qwen3.5-27b":  { "name": "Qwen/Qwen3.5-27B" },
        "qwen-vl":      { "name": "Qwen/Qwen2.5-VL-72B-Instruct" }
      }
    }
  },
  // 이 프로젝트의 MCP 서버를 도구로 등록 → 확장 재사용
  "mcp": {
    "open-claude-in-chrome": {
      "type": "local",
      "command": ["node", "/opt/openclaude/host/mcp-server.js"],
      "enabled": true
    }
  }
}
```
> opencode 설정 스키마(`provider`/`mcp` 키, AI SDK 프로바이더 명칭)는 버전에 따라 다를 수 있으니 **반입 버전 문서로 정확한 키 확인** 필수.

### 2.3 텍스트-퍼스트 강제 (소형 모델 신뢰성 핵심)

`AGENTS.md`(opencode 규칙 파일)에 **"스크린샷보다 read_page/find로 ref를 얻어 computer(ref=...)로 조작하라"** 를 못박아, 비전 없이도 동작하도록 유도. (확장의 `computer`는 좌표 대신 `ref` 클릭을 지원 → 텍스트 모델로 자동화 성립)

## 3. 모델 운용 (확정 모델 기준)

| 모델 | 역할 | 특성 | 운용 |
|---|---|---|---|
| **gpt-oss** | 텍스트 에이전트 후보 | Apache-2.0, 에이전틱/툴유즈 지향, **텍스트 전용** | PoC에서 Qwen3와 비교 |
| **Qwen3.5-27B** | 텍스트 에이전트 후보 | 툴콜 강함, **텍스트 전용** | PoC에서 gpt-oss와 비교 |
| **Qwen2.5-VL 등** | 비전(스크린샷) | 이미지 입력 | 시각 작업 시 두뇌로 전환 또는 묘사 단계 |

### 3.1 비전 구성 — "모델만 띄우면 끝"이 아님
에이전트의 **두뇌 모델이 이미지를 봐야** 스크린샷이 의미를 가집니다. 두 방식:
- **(i) 비전 모델을 두뇌로**: 시각 작업 시 opencode 모델을 VL로 전환 → 네이티브 처리. (VL이 텍스트 모델보다 에이전틱 약할 수 있음)
- **(ii) 텍스트 두뇌 + 묘사 단계**: 스크린샷을 VL이 텍스트로 묘사 → 텍스트 두뇌에 주입. (plumbing 추가)

**권장 = 하이브리드**: 기본 텍스트-퍼스트(비전 불필요), 시각이 꼭 필요한 작업만 VL 두뇌/묘사. PoC에서 "opencode가 이미지 tool-result를 모델에 실제로 전달하는지" 확인.

### 3.2 서빙 (vLLM)
- gpt-oss / Qwen3.5-27B / VL을 각각 vLLM 인스턴스로(또는 멀티모델). H100 기준 **FP8** 권장. 툴콜 파서(`--enable-auto-tool-choice --tool-call-parser …`) 모델별 설정 + **반입 vLLM 버전에서 툴콜 정상 파싱 사전 검증.** (상세는 `01` 문서 §3 참고 — 게이트웨이 부분만 OpenAI 직결로 대체)

## 4. 보안 적용 결정 (취사선택 확정)

| 안전장치 | 결정 | 구현 위치 / 동작 |
|---|---|---|
| **④ 감사 로깅** | ✅ **취함** | `mcp-server.js:430` `callTool`에서 JSONL append → 각 VDI `~/.config/open-claude-in-chrome/logs/audit-YYYY-MM-DD.jsonl`(0600). 전 도구 + js 전문 기록. 전사 통합은 SIEM 전송 별도 |
| **① 도메인 블록리스트** | ✅ **기능만**(목록 빈값) | `background.js:405` navigate에 훅 + `config.json`의 `blockedHosts`(기본 `[]` → 차단 0). 이동 감사 로깅. 나중에 도메인 추가 시 코드 변경 없이 적용 |
| **② 승인 게이트 + 자동 실행** | ✅ **취함** | `background.js:894` `update_plan`이 승인 도메인을 세션 저장 → 쓰기 도구는 승인 도메인 내 **자동 실행**, 미승인 도메인은 1회 승인 요청. 읽기 도구는 자유. 사람 승인은 **opencode 권한/모드**가 담당(Layer B), 확장 도메인 스코핑이 폭발 반경 제한(Layer A) |
| **⑤ javascript_tool** | ⛔ **통제 안 함** | 그대로 활용(폐쇄망 외부 의존 없음). 단 **감사 로깅 포함** |
| **③ 로컬 인증** | ⛔ **생략** | VDI 1인 1VM(공유 아님)이라 불필요 |

> 승인의 사람 게이트: opencode는 자체 **도구 권한/실행 모드**를 가지므로, "새 도메인 1회 승인 → 이후 자동"의 사람 확인은 opencode 권한 프롬프트로, 도메인 스코핑은 확장 측으로 — 2계층으로 동작.

## 5. 배포 (기존 VDI에 리트로핏)

VDI가 **이미 배포·사용 중**이므로, 골든 이미지를 새로 만들어 재프로비저닝할 필요는 없습니다. **이미 돌아가는 각 VDI에 설치 패키지를 1회 적용(리트로핏)** 하면 됩니다.

### 5.1 각 VDI에 설치할 컴포넌트 (1회)
1. **확장**(`extension/`)을 **고정 시스템 경로**(예: `C:\Program Files\OpenClaude\extension`)에 배치 → 브라우저에서 **Load unpacked 1회**(개발자 모드 허용 → 가능, 이후 영구 유지)
2. **Node 런타임 + `host/`**(mcp-server.js·native-host.js·vendored `node_modules`) + 네이티브 호스트 등록(Windows 레지스트리, `02` 문서 §5)
3. **opencode 설정**(`opencode.json` 커스텀 프로바이더·MCP 등록 §2.2 + `AGENTS.md`). **opencode 본체는 대부분 이미 설치됨 → 설정만 주입**, 단 **미설치 사용자 대비 동봉본을 폴백으로 포함**(install.ps1이 자동 감지: 있으면 재사용, 없으면 설치). 설치 방식(npm/CLI vs 데스크톱)은 무관, 단 데스크톱은 로컬 stdio MCP·설정경로 PoC 확인(06 §6)
4. vLLM 엔드포인트 등 설정/환경변수

> vLLM 모델은 VDI가 아니라 **중앙 GPU 서버**에 위치.

### 5.2 전달 방법 — **셀프서비스 확정** ✅
| 방법 | 적합 | 동작 |
|---|---|---|
| ~~A. 엔드포인트 관리 푸시~~ | (미채택) | — |
| **B. 셀프서비스 설치** ✅ | 기술 인력(opencode 사용자) | **사용자가 제공된 설치 스크립트를 자기 VDI에서 1회 실행** |

→ **구체 절차·설치 스크립트(`install.ps1`)·검증은 [06 문서](./06-셀프서비스-설치-가이드.md) 참조.**

> **"Load unpacked 1회"** 만 사용자가 수행하면 이후 영구 유지. 이 한 단계까지 무인화하려면 `.crx`+강제설치(Path B, `04` §1)가 유일하지만, 개발자 모드 허용이라 불필요.

### 5.3 확장 ID 일관 — 골든 이미지 없이도 유지
언팩 확장 ID는 **설치 경로 기반 해시**. 모든 VDI에 확장을 **동일한 고정 경로**로 설치하면 **ID가 전 VDI 동일** → 네이티브 호스트 `allowed_origins` 표준화. (골든 이미지가 아니어도 "고정 경로"만 지키면 됨 → **`key` 고정 불필요**.)

### 5.4 (선택) 향후 재빌드 대비
VDI를 주기적으로 재생성하는 운영이라면, **베이스 템플릿에도 동일 설치를 반영**해 두면 재빌드 후 다시 깔 필요가 없습니다(필수 아님).

### 5.5 에어갭 반입 난이도
opencode = **바이너리 1개**(npm·인터넷 불필요) → 반입 단순. (커스텀 Python 클라이언트보다 유리 — 자체 개발 제외 결정과도 합치)

## 6. 감사 로그 — 위치·포맷

- **기록 주체**: `mcp-server.js`(Node, 파일 쓰기 가능). 확장(서비스워커)은 파일 못 쓰므로, 확장 측 이벤트(차단 시도·실제 도메인)는 콘솔 + (원하면) 네이티브 메시지로 동일 파일에 추가.
- **위치**: 각 VDI `~/.config/open-claude-in-chrome/logs/audit-YYYY-MM-DD.jsonl` (Windows: `C:\Users\<user>\.config\...`), 0600, append-only.
- **1줄/이벤트(JSONL)**: `{ts, session(primary/client+id), tool, args요약(url·tabId·좌표·텍스트앞50자·js전문), result}`.
- **전사 통합**: VDI마다 분산 → 컴플라이언스 통합 감사는 filebeat/fluent-bit로 사내 SIEM(ELK/Loki) 전송(폐쇄망 내부, 코드 변경 없음).

## 7. 1일 PoC 계획

**목표**: 소형 로컬 모델(gpt-oss/Qwen3.5-27B)이 opencode로 이 확장의 브라우저 자동화를 **텍스트-퍼스트로 안정 수행**하는지 검증 — 이게 전체 프로젝트의 최대 불확실성.

1. **vLLM 기동**: gpt-oss·Qwen3.5-27B(+VL). OpenAI `/v1/chat/completions` + 툴콜 동작 확인.
2. **opencode 설치(테스트 VDI 1대)**: 커스텀 프로바이더 등록(§2.2), models.dev 외부조회 차단, 텔레메트리 off.
3. **연결**: `mcp-server.js`를 opencode MCP 서버로 등록 + 확장 Load unpacked + 네이티브 호스트 + 브라우저.
4. **텍스트-퍼스트 플로우 검증**: `tabs_context_mcp` → `navigate`(내부 사이트) → `read_page`/`find`(ref 획득) → `computer`(ref 클릭) → `form_input` → `get_page_text`.
5. **평가 항목**: 도구 올바른 선택 / ref 활용 / 다단계 진행 / 실패 복구 / 툴콜 포맷 오류율.
6. **비전 체크**: VL 모델을 두뇌로 스크린샷→클릭 1회 테스트.
7. **보안 훅 확인**: 감사 로그 기록됨 / update_plan 승인 도메인 자동실행 / 미승인 거부.

### 7.1 판정 & 대응 (커스텀 개발 제외 전제)
- **안정적** → opencode + 해당 모델 확정, 파일럿 진행.
- **불안정**(도구 오선택·다단계 실패·포맷 오류) → 대응 순서:
  1. **AGENTS.md/시스템 프롬프트 강화**(텍스트-퍼스트·도구 사용 예시 few-shot)
  2. **도구 노출 축소**(필요 도구만 enable → 소형 모델 혼란 감소)
  3. **작업 범위 축소**(짧고 명확한 플로우 위주)
  4. **더 큰 오픈모델로 교체**(말씀처럼 오픈소스면 추가 기동 가능)
  - ※ 커스텀 클라이언트 자체 개발은 **하지 않음**(결정).
